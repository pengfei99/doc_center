# Deploy opencode with llamacpp

In this tutorial, we will show you how to deploy opencode with llamacpp and run them in a totally offline environment.
Then we will use opencode to assiste a python project development.

## 1. Deploy opencode

The easiest way is to run the official installation script

```shell
curl -fsSL https://opencode.ai/install | bash
```

## 2. Deploy llama.cpp

Test your `llama.cpp` with the below command.

```shell
llama-cli \
  -m /var/lib/llama-models/Meta-Llama-3-8B-Instruct-Q4_K_M.gguf \
  --grammar '
    root   ::= "{" ws "\"service\":" ws string "," ws "\"port\":" ws number "}" ws
    ws     ::= [ \t\n\r]*
    number ::= [0-9]+
    string ::= "\"" [a-zA-Z0-9_]* "\""
  ' \
  -p "User: Output the default SSH configuration schema as JSON."
```

## 3. Choose coding model

### 3.1 Use public provider models

In this tutorial, we choose `Google Gemini` as the AI provider, because you don't need to provide your credit card.

When you run `opencode`, and get the OpenCode's TUI. Type `/connect` and press `Enter`. A list of supported providers
appears on your screen. Use the arrow keys to navigate to Google and press Enter. You’ll then be prompted for an API
key.

Google AI Studio generates Gemini API keys in a few clicks. Here are the steps:

1. Open a browser, sign in with your Google account, and navigate
   to [Google AI Studio](https://aistudio.google.com/apikey).
2. Click the first checkbox in the welcome box, then click Continue. Google AI Studio will generate the API key
   automatically.
3. Copy the key to your clipboard.
4. Return to the OpenCode interface, paste the key in the text input box, and press Enter.

## 4. Start Opencode with local llama.cpp server

### 4.1 Run the llama.cpp server as the opencode backend

Run the `Qwen3` model with llama.cpp and expose the endpoint at port `8080`

```shell
cd /opt/llama.cpp/build/bin

llama-server -m /var/lib/llama-models/Qwen3-Coder-30B-A3B-Instruct-Q4_K_M.gguf --port 8080 --host 0.0.0.0 -c 32768 -t $(nproc) --n-gpu-layers 0 --flash-attn auto --cont-batching -b 512 --no-mmap
```

options explication:

- `-c 32768`:
- `--host 0.0.0.0` :
- `-t $(nproc)` :
- `--n-gpu-layers 0`:
- `--flash-attn auto`:

Some tips:

- Use `--ctx-shift` if you hit context limits.
- Try `--no-mmap` if you have enough RAM (faster loading).
- For MoE models, newer llama.cpp builds often have good optimizations.

### 4.2 Create a custom provider in Opencode

Opencode supports many providers, you can find the complete docs [here](https://opencode.ai/docs/providers/).
As we want to use our local llama.cpp as backend, so we choose the `llama.cpp` provider.

You have two options:

- global config: `~/.config/opencode/opencode.json`
- per project config: `path/to/project/opencode.json`

In this tutorial, I choose global config.

```shell
# edit the ~/.config/opencode/opencode.json

vim ~/.config/opencode/opencode.json
```

Then add the below lines

```json
{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "llama.cpp": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "llama-server (local)",
      "options": {
        "baseURL": "http://10.50.5.75:8080/v1"
      },
      "models": {
        "Qwen3-Coder-30B-A3B-Instruct-Q4_K_M.gguf": {
          "name": "Qwen3-Coder 30B-A3B (local)",
          "limit": {
            "context": 32768,
            "output": 32768
          }
        }
      }
    }
  }
}
```

In this example:

- `llama.cpp`: is the custom provider ID. This can be any string you want.
- `npm`: specifies the package to use for this provider. Here, @ai-sdk/openai-compatible is used for any
  OpenAI-compatible API.
- `name`: is the display name for the provider in the UI.
- `options.baseURL`: is the endpoint for the local server.
- `models`: is a map of model IDs to their configurations. The model name will be displayed in the model selection list.

### Creating an AGENTS.md File

For best results, create an `AGENTS.md` file in your project root. This helps OpenCode understand:

- Your project structure
- Coding conventions
- Preferred patterns
- Technology stack

Below is a simple example of `AGENTS.md`

```shell
## Project: Build pyenv

The build-pyenv tool is a high-performance CLI tool designed to prepare Python environments for air-gapped or  restricted-network servers. It automates the process of downloading, building (if necessary), and caching Python packages files(.whl) and their dependencies.

## Tech Stack

- uv
- python >=3.11
- typer
- pip
- wheel

## Conventions
- Use functional components
- Prefer server components when possible
- Follow REST API naming conventions
- Write tests for all new features

## Structure
- /src/pbuilder - source code of the build-pyenv tool
- /tests - unit tests of the source code
- README.md - project documentation
- pyproject.toml - project dependencies and build tool specification

## Python Guidelines

- Follow PEP 8 style conventions
- Add type hints to all new and modified functions
- Prefer specific exception types over bare `except` clauses
- Use context managers for file I/O and database connections
- Write descriptive error messages that include the offending value

```

Or you can ask opencode to generate the `AGENTS.md` with the `/init` command

```shell
# 1. Navigate to your project directory
cd /path/to/your/project

# 2. run opencode
opencode

# 3. Use /models command in the TUI to select your local model.



```

### opencode basic commands

Pressing `Ctrl+P` gives you access to the list of available commands. Go ahead and explore the list to get familiar
with the commands you can use during your OpenCode sessions.

OpenCode also gives you a few shortcuts for steering a session:

- Use the `@ symbol` followed by a filename to include a file’s contents as context in your prompt.
- Start a message with an exclamation point (!) to run a shell command and have its output become part of your
  conversation.
- Use slash commands to control the session.

Type `/` in the input box to see the full list of slash commands. Here’s a summary of some useful ones:

- `/connect`: Add an AI provider or update its API key
- `/models`: List available models and switch between them
- `/init`: Create or update AGENTS.md in the current project
- `/new`: Start a new session
- `/undo`:Undo the last message and revert any file changes
- `/redo`:Restore undone messages and file changes
- `/compact`: Summarize the session to free up context window space
- `/help`: Show the help dialog
- `/exit`: Exit OpenCode

## Opencode basics

### Core Concepts: Plan vs Build Mode

OpenCode has two primary modes(plan vs build) you can switch between using the Tab key:

Plan Mode:

- Read-only - Cannot make changes to your code
- Analyzes and explores your codebase
- Suggests implementation strategies
- Perfect for understanding before modifying

Build Mode (Default)

- Full access - Can read, write, and modify files
- Executes code changes
- Runs commands
- Best for actual development work

> Pro Tip: Always start complex features in Plan mode to outline your approach, then switch to Build mode for
> implementation.
>

### Focus on a specific file

You can reference a specific file in any prompt using the `@ syntax`. Switch to Plan mode with Tab, then ask for an
analysis of any file. For example, you can enter this message
`Analyze @main.py for potential issues, bugs, and code quality problems`

## Explore What Makes OpenCode Different

There are three features that set OpenCode apart from other AI coding tools:

- Mid-session model switching: OC supports 75+ AI providers. You can connect to any of these providers at any time by
  running the `/connect` slash command, and switch to any model on the fly with the `/models` command
- Built-in language servers: OpenCode uses the `Language Server Protocol (LSP)` to interact with your code. It ships
  with more than thirty built-in language servers that run in the background when you open a source code file.
- Browser-based companion interface: start the web UI with `opencode web`, This command launches a local server on
  `127.0.0.1` and opens your default browser to the same conversational interface as the TUI:

> For Python coding, the `Pyright server` is the default and provides real-time diagnostics, including type errors,
> undefined names, and import problems. OpenCode passes this information directly to the model,
> so when you ask it to analyze your code, it already has the same diagnostic information that your editor would
> surface.

## Tips for Beginners

1. Be Specific
   ❌ "Make it better" ✅ "Refactor the handleSubmit function to use async/await and add error handling"

2. Provide Context
   ❌ "Fix the bug" ✅ "The login form throws 'undefined is not a function' when clicking submit. The error occurs in
   auth.ts line 45"

3. Use Plan Mode First
   For complex features, ask OpenCode to create a plan before implementation.

4. Leverage Image Support
   Drag and drop screenshots or mockups into the terminal—OpenCode can understand images!

5. Create Project-Specific Configs
   Different projects may need different settings. Use local opencode.json files.
