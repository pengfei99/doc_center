## 1. Why we need tokenization and embedding

Language models cannot directly understand raw human language text, instead it process:
- Tokens (Numbers)
- Vectors (A list of numbers)
- Matrices

Therefore, text must be converted into numerical representations.

Text processing pipeline

```text
Raw Text
   ↓
Preprocessing
   ↓
Tokenization
   ↓
Vocabulary Mapping
   ↓
Token IDs
   ↓
Embeddings (Vectors)
   ↓
Neural Network Input
```

## 2. Text Preprocessing

The objective of `Text Preprocessing` is to reduce variability and normalize text before tokenization.

Common preprocessing steps:
- `Lowercasing` : "Hello" → "hello"
- `Removing punctuation` : "hello!" → "hello"
- `Unicode normalization` : Standardize characters
- `Removing extra spaces`: "hello world"
- `Handling emojis` :	🙂 -> token
- `Handling URLs` : Replace with special token

For example 

```text
Hello!!! I love LLMs 😊 -> hello i love llms 😊
```

## 3. Vocabulary Construction

A vocabulary is the set of all tokens known by a model. Vocabulary Construction Process
1. Collect training text.
2. Count token frequencies.
3. Select most frequent tokens: Rare words create huge vocabularies and sparsity problems.
4. Assign token IDs.

For example, if a dataset only contains 4 word. The model trained with dataset will have a vocabulary like

| Token  | ID |
|--------|----|
| hello	 | 0  |
| world	 | 1  |
| cat    | 2  |
| dog    | 3  |

> Modern LLM vocabularies size is very large (e.g. GPT-2: 50K, Gemma-3: 256K)

### 3.1 Unknown tokens 

If the user input contains the token(word) which does not in the vocabulary, the model will be unable to answer.

> Old NLP models use a specific token(UNK) to represent unknown tokens, but this will cause information lost.
> Modern LLMs use subword tokenization to avoid unknown tokens.
> 

## 4. Types of tokenization

Based on the objective of your model, there are different types of tokenization which you can choose.

### 4.1 Character Tokenization

In this mode, each character becomes a token. For example, `hello` -> `[h,e,l,l,o]`.

Advantages:
- Tiny vocabulary
- Handles unknown words
- Works for all languages(alphabetic)

Disadvantages
- Long sequences
- Slow training
- Harder semantic learning

### 4.2 Word Tokenization

In this mode, each word becomes a token.

Advantages: 
- Shorter sequences
- Easier semantic interpretation

Disadvantages:
- Huge vocabulary
- Unknown words problem
- Poor multilingual support

### 4.3 Subword Tokenization

In this mode, words are split into meaningful sub-parts. For example, `unbelievable` -> `[un, believ, able]`

Advantages:
- Smaller vocabulary
- Handles rare words
- Better generalization
- Efficient training

> Modern LLMs primarily use subword tokenization.
> 
#### 4.3.1 Subword tokenization algorithm

`Byte Pair Encoding(BPE)` is a subword tokenization algorithm. It's originally designed for Data compression algorithm.

The core idea is `Repeatedly merge frequent token pairs.` For example, if we have three words [low,lower,lowest]. 
The first step detects `l`, 2nd step detectes `l+o`, 3rd step detectes `l+o+w`, 4th step stop merge of `low`, start a new
sequence with `e`. After the algo, we should have three tokens [low, er, est]

Benefits of BPE:

- `Efficient vocabulary`: Smaller token sets
- `Rare word handling`:	Break unknown words
- `Compression`: Reuse common subwords
- `Language flexibility`: Works across languages

## 5. Vector and embeddings

We know that tokens are still symbolic text units(represented by a numeric ID), they do not contain any semantic meaning. 
To solve this problem, we need to use another technique to represent semantic meaning between tokens.

The process of converting a token to vector is called `The Embedding Lookup`. The model which do the convert contains 
a `large embedding matrix E` of `shape (vocab_size, embedding_dimension)`.
- `vocab_size` is typically 30,000–250,000+ tokens.
- `embedding_dimension (d_model)` is usually 512, 768, 1024, or 4096.

For a token with ID i, the embedding vector is simply:$$\mathbf{v}_{token} = E[i]$$This is just a table lookup (very fast).

### 5.1 How Do These Vectors Represent Semantic Meaning?


The embedding model is trained with `massive text corpora(e.g. books, web-pages, etc.)`. During the training, the model
encode the `semantic meaning` through `the geometric relationships between vectors` in `high-dimensional space`. 

> During training, the model adjusts the `embedding matrix` so that tokens with similar meanings or functions 
> have vectors that are close to each other according to some distance metric (usually `cosine similarity`).

Mathematical view:

Two vectors $  \mathbf{u}  $ and $  \mathbf{v}  $ are semantically similar if their cosine similarity is high:$$\text{cos}(\mathbf{u}, \mathbf{v}) = \frac{\mathbf{u} \cdot \mathbf{v}}{||\mathbf{u}|| \ ||\mathbf{v}||}$$
This ranges from -1 to 1. Values near 1 = very similar meaning.
