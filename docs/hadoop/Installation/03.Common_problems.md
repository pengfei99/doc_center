# Common problems

## 1. Unable to load native-hadoop library for your platform

When your run any hadoop related commands, you will receive the below warning.

```text
Unable to load native-hadoop library for your platform... Using builtin-java classes where applicable
```

This warning is caused by `hadoop` can't find the native-hadoop library path in your execution environment.

You can specify the path with the below commands

```shell
export HADOOP_COMMON_LIB_NATIVE_DIR=$HADOOP_HOME/lib/native
export HADOOP_OPTS="$HADOOP_OPTS -Djava.library.path=$HADOOP_HOME/lib/native"
export LD_LIBRARY_PATH=$LD_LIBRARY_PATH:$HADOOP_HOME/lib/native
```

If you want to persiste this fix, you can put these three lines in `/etc/profil.d/hadoop.sh` or 
`$HADOOP_HOME/etc/hadoop/hadoop-env.sh`.

## 2. You have loaded library /.../hadoop-3.3.6/lib/native/libhadoop.so which might have disabled stack guard.

When your run any hadoop related commands, you will receive the below warning.

```text
You have loaded library /.../hadoop-3.3.6/lib/native/libhadoop.so which might have disabled stack guard. 
The vm will try to fix the stack guard now.

it's hight recommanded that you fix the library with execstack -c <libfile> or link it with -z noexecstack
```

This warning is caused by the lost symbolic link between `libhadoop.so -> libhadoop.so.1.0.0`. The `libhadoop.so`
is only a link to the real `libhadoop.so.1.0.0`. When you zip the hadoop binary with tools like zip under Windows.
The symbolic link is broken.

To fix it, you need to

```shell
cd $HADOOP_HOME/lib/native

# check the lib file
ls -lah | grep -i "libhadoop"

# if you see the libhadoop.so file has 0 byte as file size and no link to  libhadoop.so.1.0.0.
# you need to remove it and create a new one
rm libhadoop.so

ln -s libhadoop.so.1.0.0 libhadoop.so
```


## 3. while loading shared libraries: libcrypto.so.1.1 file is missing

When you run `$HADOOP_HOME/bin/container-executor --checksetup`, you may have `while loading shared libraries: libcrypto.so.1.1 file is missing`.
This error is caused by the newer Debian OS. For example in debian 13, the OpenSSL 1.1 is replaced by OpenSSL 3.x.
But the `hadoop-3.3.6` was compiled against OpenSSL 1.1.

There are two solutions:
- Install the OpenSSL 1.1 on the debian 13
- Compiled hadoop with OpenSSL 3.x.

### Install the OpenSSL 1.1 on the debian 13

The simplest solution is to download the `.deb` file and install with `dpkg -i`

You can find the download page [here](https://packages.debian.org/bullseye/amd64/openssl/download)

```shell
sudo dpkg -i openssl_1.1.1w-0+deb11u5_amd64.deb

# If dependency errors appear, fix with:
sudo apt -f install

# check the results
 /sbin/ldconfig -p | grep libcrypto.so.1.1

# expected output
libcrypto.so.1.1 (libc6,x86-64) => /lib/x86_64-linux-gnu/libcrypto.so.1.1
```


## 4. Hadoop unable to get user group

The first syndrome is you are in the group, and the group has right, you still have permission denied in your hdfs.

In your namenode, if you find the below lines in the hadoop log. It means hadoop is unable to get user group correctly

```text
org.apache.hadoop.security.ShellBasedUnixGroupsMapping: unable to return groups for user `toto`
 Can't execute shell command to get the list of group id of user `toto`. Number of group names and ids do not match
```

To confirm, you can check your user hadoop groups with the below command

```shell
# general form
hdfs groups <user-name>

# for example
hdfs groups pliu-ad

# expected output
# this means for hadoop, user pliu-ad has four groups
# if the output is empty, then it means user has 0 groups for hadoop
pliu-ad : casd hadoop casd-infra casd-ds
```

This error is caused by the group mapping config in your `core-site.xml`, by default we use `ShellBasedUnixGroupsMapping`
It retrives `gidNumber` and `groupNames` from the OS. If the number of gid and name mismatch, the mapping can't be done.
```xml
 <!-- Group resolution via OS / SSSD -->
    <property>
        <name>hadoop.security.group.mapping</name>
        <value>org.apache.hadoop.security.ShellBasedUnixGroupsMapping</value>
    </property>
```

With the below command, you can check the number of gid and name. If the number mismatch, it confirms the cause of the error.
```shell
# Count group names
id -Gn pliu-ad | wc -w

# Count group IDs
id -G pliu-ad | wc -w
```

There are two solutions:
- correct the gid and name mismatch in the OS. ()
- use `JniBasedUnixGroupsMappingWithFallback` as group mapping handler (recommended)

### 4.1 Use `JniBasedUnixGroupsMappingWithFallback` as group mapping handler

Instead of shell scripts, HDFS can use the `Java Native Interface (JNI)` to pull groups directly from the OS C-library. 
This is much more stable and handles edge cases better.

You need to modify the `core-site.xml`, replace the old `ShellBasedUnixGroupsMapping` with below line.
```xml
<property>
  <name>hadoop.security.group.mapping</name>
  <value>org.apache.hadoop.security.JniBasedUnixGroupsMappingWithFallback</value>
</property>
```

> This requires the `Hadoop native libraries (libhadoop.so)` to be correctly installed on your library path. Check 
> Section 2 on how to repair broken `libhadoop.so`
> 
> You don't need to worry about hdfs client runs on other OS type. Because hdfs client does not check user group, it 
> only forwards user id to name node. It's the name which determines user groups and permissions.
> 
If the group is from AD, and AD lags a lot. You can increase the wait time
```xml
<property>
  <name>hadoop.security.groups.cache.secs</name>
  <value>3600</value> </property>
```


### 4.2 Fix the groups in OS

If you find an GID without a name, create the group or remove that GID from the user:

As the user group is from AD imported by SSSD, so we can't remove the GID from user, we can only add the missing name

```shell
# Example: Adding a missing group name for a ghost GID
sudo groupadd -g <missing_gid> <some_name>
```
