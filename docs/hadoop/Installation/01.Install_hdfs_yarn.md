# Install hdfs(hadoop) on debian 11

**The config of the hdfs cluster in this tutorial is the minimum config to run the cluster. It should be never used
in a production environment.**

## 0. Prepare the cluster.

In this tutorial, we will use a cluster of 3 server:
- 10.50.6.103: spark-m01
- 10.50.6.104: spark-w01
- 10.50.6.105: spark-w02

### 0.1 Set server hostname

To set the hostname on each server, you can use below command

```shell
# the general form is 
hostnamectl set-hostname <new-hostname>

# in our case
hostnamectl set-hostname spark-m01

# set the value for /etc/hosts
vim /etc/hosts
# remove the default host config if there are any, it may cause the datanode unable to connect to the namenode 
127.0.0.1 spark-m01.casd.local	spark-m01

# add the below lines
10.50.6.103	    spark-m01.casd.local	spark-m01
10.50.6.104     spark-w01.casd.local    spark-w01
10.50.6.105     spark-w02.casd.local    spark-w02

# test the hostname, run below command on each server
hostname 

# test the connectivity, run below command on each server
ping spark-m01.casd.local
```

### 0.2 Create an account for hadoop related service on all three servers

```shell
# create a group called hadoop with gid 2001
sudo groupadd -g 59001 hadoop

# create a user account hadoop, the home folder is /home/hadoop, the default shell is bash
sudo useradd -g hadoop -u 2001  -r hadoop -m -d /home/hadoop --shell /bin/bash

# create a service account for hdfs, it has no home and no login shell
sudo useradd -u 2002 -g hadoop -r -M -s /sbin/nologin hdfs


# login as hadoop
sudo su -l hadoop

# add hadoop group to an existing user
# The -a (append) is essential. Otherwise, the user will be removed from any groups, not in the list.
# The -G takes a (comma-separated) list of additional groups to assign the user to.
sudo usermod -a -G groupName userName

# for example 
sudo usermod -a -G hadoop pliu
```



> The uid and gid number must be available. For the cluster which uses remote AD/Ldap server. You need to make sure there
> is no collision
> 
```shell
# check the gid/uid value of an account
getent passwd hadoop
getent group hadoop

# check if gid is used
getent group 2001
# if used, the group name with the gid will be printed

# check if uid is used
getent passwd 450
# if used, the user account will be printed 

# you can also check the reserved uid rang for system account
grep -E '^(SYS_UID_MIN|SYS_UID_MAX)' /etc/login.defs

# you can also check the reserved uid rang for system account
grep -E 'SYS_GID_MIN|SYS_GID_MAX' /etc/login.defs
```

## 2. Install and setup java
You need to install the jre and jdk version which the hadoop requires.

In this tutorial, we choose hadoop 3.3.6

### Install the package

Via package manager
```bash
sudo apt update
sudo apt upgrade

# install jre
sudo apt install default-jre

# install jdk
sudo apt install default-jdk

# test jre
java -version

# test jdk
javac -version
```

Via tarball

```shell
sudo mkdir -p /opt/java

sudo mv jdk11.tar.gz /opt/java

sudo tar -xzvf /opt/java/jdk11.tar.gz

sudo chmod -R 755 /opt/java
```

### Configure java

If you have multiple java version, you need to set up the currently used java version as the hadoop version that 
requires. 

```shell
# choose the right jre version
sudo update-alternatives --config java

# choose the right jdk version
sudo update-alternatives --config javac
```

Set **JAVA_HOME** path to the currently used jdk path. There are many ways to set the JAVA_HOME:
- In your own ~/.bashrc : the java config only works for you
- In /etc/profile.d/java.sh: The java config works for all users

We choose the second solution

```shell
# create a file to the export
sudo vim /etc/profile.d/java.sh

# add below content, you may need to change the jdk path
export JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64
export JAVA_HOME=/opt/java/jdk-11.0.2

# this is optional, because the install or update-alternatives already add the java bin path to the PATH
export PATH=$PATH:$JAVA_HOME/bin

# add the env var
source /etc/profile.d/java.sh

# test the value
echo $JAVA_HOME
```

You can get the latest hadoop release from this page. https://hadoop.apache.org/releases.html

## 4. Configure the hadoop cluster

As we know, the hadoop cluster has three parts:
- hdfs
- yarn
- mapreduce

Hadoop can be configured to run in a `single node` or `multi-node` cluster. In this tutorial, we only shows the multi
node cluster.

All the configuration files which we edit below are already existed. We only need to add few lines.

> All Hadoop processes run on a Java Virtual Machine (JVM).

### 4.1 Configure hdfs

There is a doc on how to configure hdfs written by cloudera
https://docs.cloudera.com/runtime/7.2.0/hdfs-overview/topics/hdfs-sizing-namenode-heap-memory.html

#### hadoop-env.sh

This `hadoop-env.sh` file is used to set various `Environment variables` for Hadoop's execution. 

Some important conf options: 
- **JAVA_HOME**: You can specify the path to the Java home directory if you want to use a specific Java installation.

- **HADOOP_OPTS**: Additional Java options to be applied to all Hadoop commands.

- **HADOOP_NAMENODE_OPTS, HADOOP_DATANODE_OPTS**, and similar variables: These allow you to set JVM options specific to different Hadoop daemon processes like the NameNode, DataNode, etc.

- **HADOOP_HEAPSIZE_MAX/MIN**: Sets the maximum/minimum heap size for the JVM, usually in MB.

- **HADOOP_LOG_DIR and HADOOP_MAPRED_LOG_DIR**: Define the directories where log files are stored.

- **HADOOP_CLASSPATH**: Allows you to specify additional classpath elements if needed.

- **PATH**: It adds Hadoop's bin/ directory to the system PATH.


```shell
vim etc/hadoop/hadoop-env.sh

# In our case, we only add below lines
export JAVA_HOME=/opt/java/jdk-11.0.2

export HADOOP_HEAPSIZE=4096 # Set heap size to 4 GB for both NameNode and DataNode

# Set JVM options for NameNode
export HADOOP_NAMENODE_OPTS="-Xmx2048m -Xms2048m" # Set initial and maximum heap size for NameNode

# Set JVM options for DataNode
export HADOOP_DATANODE_OPTS="-Xmx1024m -Xms1024m" # Set initial and maximum heap size for DataNode
```

> Xmx is the max heap memory, Xms is the initial head size

### 4.2 core-site.xml

The conf file `core-site.xml` contains settings that are `common to all Hadoop components`
(e.g.HDFS, YARN, MapReduce, Spark, Hive, etc.).

Key properties you’ll commonly see:
- `fs.defaultFS`: Default filesystem URI (tells everyone where HDFS is). Example value: `hdfs://namenode-host:8020`
- `hadoop.tmp.dir`: Base temporary directory (very important). Example value: `/app/hadoop/tmp`
- `io.compression.codecs`: Compression codecs available. Example value: `Snappy, Zlib, LZ4, etc.`
- `hadoop.proxyuser.*`: For proxy users (Hue, Oozie, Spark, etc.)
- `fs.trash.interval`: How long deleted files stay in trash. Example value: 4320 (3 days)

```shell
vim etc/hadoop/core-site.xml
```

This conf is common for both the `datanode and namenode`.
```xml
<configuration>
  <property>
    <name>fs.defaultFS</name>
    <value>hdfs://master:9000</value>
  </property>
  <property>
    <name>hadoop.tmp.dir</name>
    <value>file:///opt/hadoop/data/tmp</value>
  </property>
  <property>
    <name>io.file.buffer.size</name>
    <value>16384</value>
  </property>
</configuration>
```

here we have three config:
- fs.defaultFS: indicates where is the namenode
- hadoop.tmp.dir: defines where to put temp file of hdfs and mapreduce. The default value is `/tmp/hadoop-${user.name}`, this may be deleted by system clean. So we recommend you to set up another path for it.
- io.file.buffer.size: defined the buffer size during read/write of hdfs. The default value is `4096 bytes (4 KB)`. We set it to `16384 bytes (16 KB)` to increase speed. but it will consume more memory.

### 4.3 hdfs-site.xml

The `hdfs-site.xml` file controls everything related to HDFS (NameNode, DataNode, JournalNode, etc.).
Key properties you will commonly see:
- `dfs.replication`: Default replication factor (usually 3)
- `dfs.namenode.name.dir`: Where NameNode stores metadata (fsimage + edits)
- `dfs.datanode.data.dir`: Where DataNodes store actual blocks
- `dfs.blocksize`: Default block size (128MB or 256MB in modern clusters)
- `dfs.namenode.http-address`: Web UI of NameNode
- `dfs.ha.*`: High Availability settings (for HA clusters)
- `dfs.permissions.enabled`: Whether permission checking is on

> **dfs.datanode.data.dir/dfs.data.dir(old version)** : Determines where on the local filesystem an DFS data node should store its blocks. If this is a 
 comma-delimited list of directories, then data will be stored in all named directories, typically on different devices. 
 Directories that do not exist are ignored.

For the `hadoop 3.3.*`, you can use the below conf

For the `namenode`, edit `hdfs-site.xml` add the following lines

```shell
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
     <property>
        <name>dfs.replication</name>
        <value>2</value>
    </property>
    
    <property>
        <name>dfs.namenode.name.dir</name>
        <value>file:///opt/hadoop/data/hdfs/namenode</value>
    </property>
   
    <property>
    <name>dfs.namenode.checkpoint.dir</name>
    <value>file:///opt/hadoop/data/hdfs/secondary</value>
    </property>
</configuration>
```

> `dfs.namenode.checkpoint.dir` defines the `checkpoint daemon(secondary namenode)` where to store its local files.

For the `datanode`, edit `hdfs-site.xml` add the following lines
```shell
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
     <property>
        <name>dfs.replication</name>
        <value>2</value>
    </property>
    
    <property>
        <name>dfs.datanode.data.dir</name>
        <value>file:///opt/hadoop/data/hdfs/datanode</value>
    </property>
</configuration>    
```

> Create the required data folders if they do not exit yet :
> - /opt/hadoop/data/hdfs/namenode
> - /opt/hadoop/data/hdfs/datanode
> - /opt/hadoop/data/hdfs/secondary

#### Why we run a secondary name node

The name `secondary namenode` is confusing. Because, it's not a standby replication of the namenode. Its real job is 
`checkpointing metadata`. We should call it a `checkpoint` daemon.

The name node stores `fsimage(full filesystem snapshot)` and `edits(transaction log)`.
Each file operation on hdfs will generate a new line in `edits`, if we don't create checkpoint for `edits` log. The `edits` log
`file size will be uncontrollably`. At restart, the NameNode must replay the entire `edits log`. If it becomes huge:
- startup takes hours
- memory pressure increases
- risk of corruption rises
- recovery becomes painful

The Secondary NameNode periodically:
- downloads fsimage + edits
- merges them
- sends back a compact checkpoint

> This will keep the namenode metadata healthy.

### 4.4 yarn-site.xml

The `yarn-site.xml` configures the `resource management layer (YARN)`.

Key properties you’ll commonly see: 
- `yarn.resourcemanager.hostname`: Hostname of ResourceManager
- `yarn.nodemanager.aux-services`: Usually mapreduce_shuffle
- `yarn.nodemanager.resource.memory-mb`: Total memory available on each Node Manager
- `yarn.scheduler.minimum-allocation-mb`: Minimum container size
- `yarn.log-aggregation-enable`: Enable log aggregation
- `yarn.acl.enable`: Security / queue ACLs

Below is an example of `yarn-site.xml` on Master Node (ResourceManager settings)

```xml

<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <property>
        <name>yarn.resourcemanager.hostname</name>
        <value>master</value>
        <description>Hostname of the ResourceManager.</description>
    </property>
    <property>
        <name>yarn.nodemanager.aux-services</name>
        <value>mapreduce_shuffle</value>
        <description>Auxiliary services for NodeManager.</description>
    </property>
    <property>
        <name>yarn.nodemanager.aux-services.mapreduce.shuffle.class</name>
        <value>org.apache.hadoop.mapred.ShuffleHandler</value>
    </property>
    <property>
        <name>yarn.scheduler.minimum-allocation-mb</name>
        <value>1024</value>
        <description>Minimum container memory allocation.</description>
    </property>
    <property>
        <name>yarn.scheduler.maximum-allocation-mb</name>
        <value>4096</value>
        <description>Maximum container memory (12GB per worker).</description>
    </property>
    <property>
        <name>yarn.nodemanager.resource.memory-mb</name>
        <value>4096</value>
        <description>Total memory for NodeManager (if master also acts as worker; adjust if not).</description>
    </property>
</configuration>
```

`yarn-site.xml` on Worker Nodes (NodeManager settings)


### 4.5 mapred-site.xml

The `mapred-site.xml` configures the MapReduce framework.

Key properties you’ll commonly see: 
- `mapreduce.framework.name`:  must be set to yarn
- `mapreduce.map.memory.mb`
- `mapreduce.reduce.memory.mb`
- `mapreduce.jobhistory.address`
- `yarn.app.mapreduce.am.env`

```shell
<configuration>
    <property>
        <name>mapreduce.framework.name</name>
        <value>yarn</value>
    </property>
    <property>
    <name>mapreduce.jobhistory.address</name>
    <value>master:10020</value>
  </property>

  <property>
    <name>mapreduce.jobhistory.webapp.address</name>
    <value>master:19888</value>
  </property>
</configuration>
```



## 5. Create master and worker files

The `workers (or slaves in old versions)` file list of all `DataNodes/NodeManagers (one hostname per line)` url.
The **workers** file is used by startup scripts of the hadoop cluster to identify the data node address. It's located 
at `~/hadoop/etc/hadoop/workers`

```shell
# we put the data node ip address
10.50.5.68
10.50.5.69
```

The **master file is no longer needed since hadoop 2.7**. In old version, we use it to list name node location  
at `~/hadoop/etc/hadoop/masters`

```shell
# we put the name node ip address
10.50.5.67
```

## 6. start/stop the cluster

### 6.1 Format the NameNode

When starting HDFS for the first time, the NameNode must be formatted exactly once before HDFS can start.

```shell
# stop the cluster if it's already started
./stop-dfs.sh

# format the namenode
hdfs namenode -format

# expected output
Storage directory ... has been successfully formatted
```

### 6.2 Useful commands for starting the cluster

The hadoop provides a list of script which allows us to start and stop the cluster easily. You can find these scripts
in `/opt/hadoop/sbin`

```shell
# start the hdfs cluster
./sbin/start-dfs.sh

# you should see below outputs
Starting namenodes on [k8s-master]
Starting datanodes
Starting secondary namenodes [k8s-master]

# you can check the running java process
jps

# in master node, you should see
138517 NameNode
138757 SecondaryNameNode
143035 Jps

# in data node, you should see
88904 DataNode
91423 Jps

# start the yarn 
./sbin/start-yarn.sh

# you should see one more process in jps
ResourceManager
```

## 9. Useful url

- **hdfs UI**: (http://10.50.5.67:9870): monitor hdfs cluster status, storage usage and data lists
- **Yarn resource manager UI**: (http://master:8088): monitor cluster total and used resource for running jobs, job list and different job queues
- **Yarn node manager UI**: (http://worker:8042): monitor applications and containers running on this node,
- **spark UI** (http://10.50.5.67:8088/proxy/<application_id>): monitor current running spark jobs progress and resource utilisation
- **spark history server** : (http://10.50.5.67:18080): Monitor the stats of terminated spark jobs

## Other useful configuration file

Depends on which property you want to use in hadoop, you may want to edit the below config files:
- `capacity-scheduler.xml`: Queue configuration when using Capacity Scheduler (most common)
- `fair-scheduler.xml`: Queue config when using Fair Scheduler
- `log4j.properties`: Logging levels for all daemon
- `sssl-server.xml` / `ssl-client.xml`: SSL/TLS configuration
- `httpfs-site.xml`: Configuration for HttpFS (REST access to HDFS)
- `kms-site.xml`: Key Management Server (for encryption at rest)
