    Optimize hdfs¶

    The default config of hdfs can't be used in the production environment. The below config need to be adjusted to make your cluster work correctly.

    The official hadoop doc can be found here. Choose the version which you use, and check the config parameters. For example, we use 3.3.6, the config params for hdfs-site.xml

    1. NameNode optimization¶

    In a Hadoop 3.3.6 HDFS cluster, you can tune three areas easily for a better performance:

    • NameNode memory allocation
    • Handler/concurrency limits
    • Protection against user overload (quotas & limits)

    1.1 NameNode memory allocation¶

    The NameNode stores all filesystem metadata in memory (files, blocks, permissions, ACLs).

    Rough estimation:

    • 150–200 bytes per block
    • 150 bytes per file/inode

    For Example:

    • 10 million blocks → ~2 GB metadata
    • With JVM overhead → ~4–6 GB

    For a 16 GB machine, a common configuration is to set NameNode Heap: 8–10 GB

    You can add the below lines in hadoop-env.sh

    export HDFS_NAMENODE_OPTS="
-Xms8g
-Xmx10g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:+ParallelRefProcEnabled
"

    1.2 Increase NameNode RPC Handler Threads¶

    Each client request (read, write, list, permission check) uses an RPC handler.

    The default NameNode RPC Handler Threads number is dfs.namenode.handler.count = 10

    In a prod environment, you can imagine 1 thread for each user. If we have 40 users in total, we should set the handler number to 40. Or we can use the below formula to calculate the number.

    In hdfs-site.xml, you can add the following lines 

    <property>
  <name>dfs.namenode.handler.count</name>
  <value>40</value>
    <description>process handles client read/write/list/acl_check request </description>
</property>

<property>
  <name>dfs.namenode.service.handler.count</name>
  <value>20</value>
<description>process handles DataNode → NameNode communications</description>
</property>


    import math
# N is the user number
print int(math.log(N) * 20)
    | N users | log(N) | |---------|---------| | 10 | 2.30 | | 40 | 3.69 | | 100 | 4.60 | | 1000 | 6.90 |

    This will prevent the explosion of the thread number as user increase.

    Increase Client ipc Limits¶

    In core-site.xml, you can put the below config. 

    <property>
  <name>ipc.server.max.connections</name>
  <value>2000</value>
    <description>the default value is `0`, which mean no limits. </description>
</property>

    This prevents: - Socket Overhead: Each connection consumes a small amount of memory and a file descriptor. With 16GB of RAM, 2000 connections is well within the limits of what your OS can handle. - DDOS attack on the NameNode by opening infinite sockets

    Enable NameNode Caching (optional but useful)¶

    The below config are in hdfs-site.xml

    <property>
    <name>dfs.namenode.list.cache.directives.num.responses</name>
    <value>150</value>
    <description>The value controls pagination when querying HDFS caching directives. The default value is 100</description>
</property>

    It sets the maximum number of cache directives returned in a single "batch" or response. If there are more directives than this limit, the client must make multiple sequential calls (using a "marker" or "cursor") to get the full list.

    The NameNode’s biggest enemy is Stop-the-World Garbage Collection (GC) pauses.

    If you set this value too high (e.g., 10000), and a user requests a list, the NameNode has to: - Serialize a massive object into memory. - Send it over the network. - Risk triggering a GC pause because of the sudden spike in transient memory usage.

    Other cache parameters need to check

    • dfs.namenode.path.based.cache.refresh.interval.ms: The recommanded value is 30000 (30s). It controls how often the NN scans for cache changes.
    • dfs.datanode.max.locked.memory: It controls how much RAM each DataNode can use for caching.

    Important Note: HDFS Caching requires the OS limit memlock to be increased on the DataNodes, or the cache will fail to "lock" into memory.

    1.3 Protect NameNode From Metadata Explosion¶

    40 users can create millions of tiny files. Small files kill NameNode memory.

    Use directory quotas.

    hdfs dfsadmin -setQuota 200000 /home/alice
hdfs dfsadmin -setQuota 200000 /home/bob
    This limits inode count.

    We can also set space quota:

    hdfs dfsadmin -setSpaceQuota 5T /home/alice

    Appendix. Check hardware (network, diskIO)¶

    Check Disk IO performance¶

    Here we use a tool called fio to check the disk read/write speed

    sudo apt install fio

# check sequential read speed
sudo fio -filename=/tmp/test.log -direct=1 -iodepth 1 -thread -rw=read -ioengine=psync -bs=16k -size=2G -numjobs=10 \
-runtime=60 -group_reporting -name=test_r

# check sequential write speed
sudo fio -filename=/tmp/test.log -direct=1 -iodepth 1 -thread -rw=write -ioengine=psync -bs=16k -size=2G -numjobs=10 \
-runtime=60 -group_reporting -name=test_w

# check random write
sudo fio -filename=/tmp/test.log -direct=1 -iodepth 1 -thread -rw=randwrite -ioengine=psync -bs=16k -size=2G -numjobs=10 \
-runtime=60 -group_reporting -name=test_randw

# check mix random read/write

sudo fio -filename=/tmp/test.log -direct=1 -iodepth 1 -thread -rw=randrw -ioengine=psync -bs=16k -size=2G -numjobs=10 \
-runtime=60 -group_reporting -name=test_rand_r_w
