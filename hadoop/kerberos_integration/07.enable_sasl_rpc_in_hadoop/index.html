    Enable sasl rpc in hadoop cluster¶

    When you integrate kerberos into Hadoop cluster, the datanode will be started in secure mode. To run data node in secure mode, you need to configure one of following options(not both): - privileged resources using the jsvc program: Legacy solution, not recommended - SASL RPC data transfer with non-privileged ports: recommended solution.

    For more details, you can visit this page.

    As a result, we must set up all the required config to run SASL.

    Generate certificate and key store for name node¶

    In this tutorial, we choose to use an internal CA. Then we use the CA certificate to signe the certificate for the three nodes.

    # generate root CA certificate
openssl genrsa -out ca.key 4096
openssl req -x509 -new -nodes -key ca.key -sha256 -days 3650 -out ca.pem


# generate private key and csr for the name node
keytool -genkeypair \
  -alias nn \
  -keyalg RSA \
  -keysize 2048 \
  -storetype PKCS12 \
  -keystore keystore.p12 \
  -validity 365 \
  -storepass casd2026 \
  -keypass casd2026 \
  -dname "CN=deb13-spark1.casdds.casd, OU=Hadoop, O=Cluster, C=FR"

# export the csr from the keystore
keytool -certreq \
  -alias nn \
  -keystore keystore.p12 \
  -storetype PKCS12 \
  -file nn.csr \
  -storepass casd2026

# sign the csr with root ca
# openssl x509: indicates using x509 subcommand to handle x509 certificate
# -req -in nn.csr: the input is the nn.csr file.
# -CA ca.pem: Use ca.pem as the CA certificate (the issuer)
# -CAkey ca.key: Use the private key of the CA (ca.key) to perform the signing.
# -CAcreateserial: Create a new serial number file (ca.srl) if it doesn't exist, and use/increment the serial 
# number for this certificate. This ensures every certificate signed by this CA gets a unique serial.
# -out nn.crt: Write the signed certificate to nn.crt.
# -days 1365: Make the certificate valid for 1365 days (~3.74 years). After this it will expire.
# -sha256: Use SHA-256 as the signature algorithm (secure and widely supported; better than the old SHA-1).
# -extfile nn.ext: Read certificate extensions from the file `nn.ext`.
# -extensions v3_req: Use the section named [v3_req] to copy the extensions into the final certificate inside the `nn.ext` file.
openssl x509 -req -in nn.csr -CA ca.pem -CAkey ca.key -CAcreateserial -out nn.crt -days 1365 -sha256 -extfile nn.ext -extensions v3_req

# import root ca to the keystore
keytool -import \
  -alias cluster-ca \
  -file ca.pem \
  -keystore keystore.p12 \
  -storetype PKCS12 \
  -storepass casd2026

# import the name node certificate into the keystore
keytool -import \
  -alias nn \
  -file nn.crt \
  -keystore keystore.p12 \
  -storetype PKCS12 \
  -storepass casd2026

# you can check the keystore conetent with 
keytool -list -v -keystore keystore.p12 -storetype PKCS12

    The nn.ext file specifies the TLS/SSL certificates (X.509v3) extensions specification. Below is an example of nn.ext. Because the modern TLS/SSL certificates (X.509v3) rely on extension to allow CA add extra controls. Without these extensions, the certficate may not work(not accepted by the browser).

    [v3_req]
basicConstraints = CA:FALSE
keyUsage = digitalSignature, keyEncipherment
extendedKeyUsage = serverAuth, clientAuth
subjectAltName = @alt_names

[alt_names]
DNS.1 = deb13-spark1.casdds.casd
    - subjectAltName = @alt_names: The most important extension in practice. It specifies all the hostnames and IP addresses this certificate is valid for (e.g., DNS.1 = example.com, DNS.2 = www.example.com, IP.1 = 192.168.1.10). The Common Name (CN) in the CSR is largely ignored by modern clients. SubjectAltName (SAN) is what browsers and clients actually use. Without it, you often get warnings like "hostname mismatch" or the certificate is rejected. - basicConstraints = CA:FALSE: Explicitly states that this is not a CA certificate. Prevents the certificate from being used to sign other certificates (important for security — stops someone from turning your server cert into a rogue CA). - keyUsage = digitalSignature, keyEncipherment: Restricts what the public key in this certificate is allowed to do. digitalSignature: Used for signing (e.g., in TLS handshake). keyEncipherment: Used for encrypting keys (e.g., RSA key exchange in TLS). This makes the certificate more secure and compliant. - extendedKeyUsage = serverAuth, clientAuth: Tells applications the intended purpose of the certificate. serverAuth: This certificate can be used by a server (HTTPS, TLS server). clientAuth: This certificate can also be used for client authentication (mutual TLS). Many applications (especially browsers and modern servers) check this.

    The file dn2.ext should look like:

    [v3_req]
basicConstraints = CA:FALSE
keyUsage = digitalSignature, keyEncipherment
extendedKeyUsage = serverAuth, clientAuth
subjectAltName = @alt_names

[alt_names]
DNS.1 = deb13-spark3.casdds.casd

    After the above command, we should have three keystore files - keystore-nn.p12 - keystore-dn1.p12 - keystore-dn2.p12

    Now, in each node, we create a folder /etc/security/ssl, then place the keystore in it.

    Edit the ssl-server.xml¶

    In all nodes, we need to edit the $HADOOP_HOME/etc/hadoop/ssl-server.xml

    <?xml version="1.0"?>
<?xml-stylesheet type="text/xsl" href="configuration.xsl"?>

<configuration>

    <property>
        <name>ssl.server.truststore.location</name>
        <value>/etc/security/ssl/keystore-nn.p12</value>
        <description>Truststore to be used by NN and DN. Must be specified.
        </description>
    </property>

    <property>
        <name>ssl.server.truststore.password</name>
        <value>casd2026</value>
        <description>Optional. Default value is "".
        </description>
    </property>

    <property>
        <name>ssl.server.truststore.type</name>
        <value>PKCS12</value>
        <description>Optional. The keystore file format, default value is "jks".
        </description>
    </property>

    <property>
        <name>ssl.server.truststore.reload.interval</name>
        <value>10000</value>
        <description>Truststore reload check interval, in milliseconds.
            Default value is 10000 (10 seconds).
        </description>
    </property>

    <property>
        <name>ssl.server.keystore.location</name>
        <value>/etc/security/ssl/keystore-nn.p12</value>
        <description>Keystore to be used by NN and DN. Must be specified.
        </description>
    </property>

    <property>
        <name>ssl.server.keystore.password</name>
        <value>casd2026</value>
        <description>Must be specified.
        </description>
    </property>

    <property>
        <name>ssl.server.keystore.keypassword</name>
        <value>casd2026</value>
        <description>Must be specified.
        </description>
    </property>

    <property>
        <name>ssl.server.keystore.type</name>
        <value>PKCS12</value>
        <description>Optional. The keystore file format, default value is "jks". </description>
    </property>

    <property>
        <name>ssl.server.exclude.cipher.list</name>
        <value>TLS_ECDHE_RSA_WITH_RC4_128_SHA,SSL_DHE_RSA_EXPORT_WITH_DES40_CBC_SHA,
            SSL_RSA_WITH_DES_CBC_SHA,SSL_DHE_RSA_WITH_DES_CBC_SHA,
            SSL_RSA_EXPORT_WITH_RC4_40_MD5,SSL_RSA_EXPORT_WITH_DES40_CBC_SHA,
            SSL_RSA_WITH_RC4_128_MD5
        </value>
        <description>Optional. The weak security cipher suites that you want excluded
            from SSL communication.
        </description>
    </property>

</configuration>
